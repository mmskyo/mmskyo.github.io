## 즐겨찾기 기능 복습 노트

---

### 🧱 먼저 큰 그림 (레고 비유)

즐겨찾기 기능은 **4개의 레이어**로 구성돼 있어요. 각 레이어는 딱 자기 역할만 합니다.

```
👤 사용자가 버튼 누름
       ↓
🖥️  Fragment      → 화면 담당. 버튼 클릭 감지
       ↓
🧠  ViewModel     → 두뇌. "뭘 해야 하지?" 판단
       ↓
📦  Repository    → 실무자. 서버/DB에 실제로 일 시킴
       ↓
🗄️  DB / 서버    → 데이터 저장소
```

> **핵심 규칙**: 각 레이어는 바로 아래 레이어하고만 대화해요. Fragment가 DB를 직접 건드리면 ❌

---

### 🗄️ LAYER 1 — 데이터 저장소

#### Room DB (로컬 창고)

폰 안에 있는 미니 데이터베이스예요.

```kotlin
@Entity(tableName = "favorites_cache")
data class FavoriteCacheEntity(
    @PrimaryKey val favoriteId: String,  // 서버가 준 ID
    val url: String,
    val title: String?,
    val riskLevel: String,
    val addedAt: String
)
```

> **왜 로컬 DB에 저장해?** 서버 없이도 즐겨찾기 목록을 빠르게 보여주려고. 앱 켤 때마다 서버 기다리면 느리니까.

#### DAO (창고 직원)

DB에 명령 내리는 인터페이스예요.

```kotlin
@Dao
interface FavoriteCacheDao {
    @Query("SELECT * FROM favorites_cache ORDER BY addedAt DESC")
    fun getAll(): Flow<List<FavoriteCacheEntity>>   // 목록 구독

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insert(entity: FavoriteCacheEntity) // 추가

    @Query("DELETE FROM favorites_cache WHERE url = :url")
    suspend fun deleteByUrl(url: String)            // 삭제

    @Query("SELECT * FROM favorites_cache WHERE url = :url LIMIT 1")
    suspend fun getByUrl(url: String): FavoriteCacheEntity? // 조회
}
```

> **Flow가 뭐야?** "데이터가 바뀔 때마다 자동으로 알려줘" 하는 구독 시스템. 유튜브 알림 구독이랑 같은 개념이에요.

---

### 📦 LAYER 2 — Repository (실무 담당자)

서버 API 호출 + 로컬 DB 저장을 한 곳에서 처리해요.

```kotlin
@Singleton
class BookmarkRepository @Inject constructor(
    private val apiService: ApiService,
    private val favoriteCacheDao: FavoriteCacheDao
) {
```

**메서드 3개만 기억하면 됩니다:**

```kotlin
// 1. 서버 → 로컬 DB 동기화
suspend fun syncFromServer() {
    val response = apiService.getBookmarks()       // 서버에 요청
    if (response.isSuccessful) {
        val entities = response.body()?.items?.map { ... }
        favoriteCacheDao.clearAll()                // 기존 삭제
        favoriteCacheDao.insertAll(entities)       // 새로 저장
    }
}

// 2. 즐겨찾기 추가 (서버 먼저, 그 다음 로컬)
suspend fun add(url: String, title: String?, riskLevel: String) {
    val response = apiService.addBookmark(AddBookmarkRequest(url, title))
    if (response.isSuccessful) {
        val bookmarkId = response.body()!!.bookmarkId  // 서버 ID 받기
        favoriteCacheDao.insert(FavoriteCacheEntity(favoriteId = bookmarkId, ...))
    }
}

// 3. 즐겨찾기 삭제 (로컬 ID로 서버 ID 찾아서 삭제)
suspend fun remove(url: String) {
    val entity = favoriteCacheDao.getByUrl(url) ?: return
    apiService.deleteBookmark(entity.favoriteId)  // 서버 삭제
    favoriteCacheDao.deleteByUrl(url)             // 로컬 삭제
}
```

> **왜 서버 먼저야?** 서버가 실패하면 로컬에도 저장하면 안 되니까. 반대로 하면 서버 실패해도 로컬엔 있어서 데이터가 어긋남.

### 🧠 LAYER 3 — ViewModel (두뇌)

UI 상태를 관리하고 Repository에 일을 시켜요.

#### FavoritesViewModel (목록 화면용)

```kotlin
@HiltViewModel
class FavoritesViewModel @Inject constructor(
    private val bookmarkRepository: BookmarkRepository
) : ViewModel() {

    // Flow → StateFlow 변환 (화면이 구독할 수 있게)
    val favorites: StateFlow<List<FavoriteCacheEntity>> =
        bookmarkRepository.getAll()
            .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), emptyList())

    init {
        // 화면 처음 열릴 때 서버 동기화
        viewModelScope.launch { bookmarkRepository.syncFromServer() }
    }

    fun delete(url: String) {
        viewModelScope.launch { bookmarkRepository.remove(url) }
    }
}
```

#### ResultViewModel (스캔 결과 화면용)

```kotlin
@HiltViewModel
class ResultViewModel @Inject constructor(
    private val bookmarkRepository: BookmarkRepository
) : ViewModel() {

    private val _isFavorite = MutableStateFlow(false)
    val isFavorite = _isFavorite.asStateFlow()

    // 현재 URL이 즐겨찾기인지 확인
    fun checkFavorite(url: String) {
        viewModelScope.launch {
            _isFavorite.value = bookmarkRepository.isFavorite(url)
        }
    }

    // 즐겨찾기 토글 (있으면 삭제, 없으면 추가)
    fun toggleFavorite(url: String, title: String?, riskLevel: String) {
        viewModelScope.launch {
            if (_isFavorite.value) bookmarkRepository.remove(url)
            else bookmarkRepository.add(url, title, riskLevel)
            _isFavorite.value = !_isFavorite.value
        }
    }
}
```

> **StateFlow가 뭐야?** ViewModel이 가진 "현재 상태"예요. Fragment가 구독하면 값이 바뀔 때마다 자동으로 화면이 업데이트돼요.

### 🖥️ LAYER 4 — Fragment (화면)

ViewModel을 구독하고, 사용자 액션을 ViewModel에 전달해요.

```kotlin
// ViewModel 연결
private val viewModel: FavoritesViewModel by viewModels()

override fun onViewCreated(...) {
    // RecyclerView 설정
    binding.rvFavorites.adapter = favoritesAdapter
    binding.rvFavorites.layoutManager = LinearLayoutManager(requireContext())

    // ViewModel 구독 → 데이터 바뀌면 자동 업데이트
    viewLifecycleOwner.lifecycleScope.launch {
        viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
            viewModel.favorites.collect { list ->
                if (list.isEmpty()) {
                    binding.layoutEmpty.visibility = View.VISIBLE
                    binding.rvFavorites.visibility = View.GONE
                } else {
                    binding.layoutEmpty.visibility = View.GONE
                    binding.rvFavorites.visibility = View.VISIBLE
                    favoritesAdapter.submitList(list)
                }
            }
        }
    }
}
```

### 🔄 전체 흐름 한눈에 보기

**즐겨찾기 탭 열 때:**

```
Fragment 열림
→ ViewModel init 실행
→ bookmarkRepository.syncFromServer()
→ 서버 GET 요청
→ 응답 받아서 로컬 DB에 저장
→ DB가 Flow로 변경 알림
→ StateFlow 업데이트
→ Fragment의 collect 실행
→ 화면 자동 갱신
```

**즐겨찾기 추가할 때:**

```
사용자가 ★ 버튼 클릭
→ Fragment: viewModel.toggleFavorite(url, title, riskLevel)
→ ViewModel: bookmarkRepository.add()
→ Repository: 서버 POST 요청
→ 서버가 bookmarkId 응답
→ 로컬 DB insert
→ Flow 자동 방출
→ 화면 자동 갱신
```

### 🔑 처음부터 구현할 때 순서

혼자 만들 때 이 순서대로 하면 됩니다:

```
1. Entity 만들기         → DB에 뭘 저장할지 정의
2. DAO 만들기            → DB 명령어 작성
3. AppDatabase 등록      → entities에 추가
4. Repository 만들기     → API + DAO 연결
5. ViewModel 만들기      → StateFlow + 함수
6. Fragment 만들기       → collect + 어댑터 연결
7. 레이아웃 XML 만들기   → 화면 구성
8. 네비게이션 등록       → nav_graph.xml
```

> **막힐 때 체크리스트**
> 
> - Flow가 안 오면? → DAO 반환 타입이 `Flow<>`인지 확인
> - 화면이 안 바뀌면? → `repeatOnLifecycle` 빠진 건 아닌지 확인
> - 서버 요청이 안 가면? → AuthInterceptor가 토큰 붙이는지 Logcat 확인
> - 삭제가 서버에 안 되면? → `favoriteId`가 서버에서 받은 ID인지 확인
